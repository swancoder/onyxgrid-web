# Onyx Gateway — Technical Project Digest

## Overview & Current State

**Onyx Gateway** is a lightweight Command and Control (C2) gateway designed for edge-deployable passive sensor mesh networks. It aggregates MQTT telemetry from distributed sensor nodes, maintains state in a persistent database, and broadcasts tactical network information via UDP multicast while streaming real-time updates to a web-based operations console.

**Technology Readiness Level: TRL 4** (Alpha MVP, technology validated in lab environment with simulated sensor payloads).

The system is currently operational with synthetic threat and node telemetry generated by an embedded simulator. Production deployment requires integration with real edge-deployed Onyx Grid sensor nodes.

---

## Implemented Functionality

### Core Ingestion & Persistence
- **MQTT Broker Integration:** Consumes sensor heartbeats and threat alerts over MQTT with mutual TLS authentication (mTLS, TLS 1.3, port 8883).
- **Database Layer:** PostgreSQL 16 with Flyway schema management; tracks node state (identity, geolocation, last-seen timestamp, lifecycle), threat events (classification, azimuth, confidence, speed), and spatial perimeters (geofenced regions).
- **Event Normalization:** Validates and clamps telemetry payloads; speed values constrained to 0–150 km/h; azimuth normalized to 0–359°.

### Tactical Network Broadcasting
- **Cursor on Target (CoT) Multicast:** Generates and broadcasts standard CoT XML over UDP multicast (239.2.3.1:6969) for integration with tactical operations software (e.g., ATAK).
- **Real-Time State Streaming:** WebSocket endpoint (`/ws/state`) pushes node state changes and threat event detections to authenticated clients in JSON format.

### Web-Based Operations Console
- **Offline-Capable UI:** MapLibre GL JS with vector tile rendering (Protomaps PMTiles), no external tile server dependency.
- **Node & Threat Visualization:** 
  - Real-time node markers (color-coded: green=ALIVE, gray=OFFLINE, red=THREAT).
  - Directional threat arrows (SVG triangles) rotated by azimuth bearing.
  - Threat metadata displayed on marker click: classification, direction (cardinal + azimuth degrees), speed, confidence percentage.
- **Geospatial Context:** Displays configurable perimeter zones (polygons) overlaid on map.
- **Alert Feed:** Right-aligned chronological threat log with type, direction, speed, and confidence metrics.
- **Dashboard Controls:** Sidebar with perimeter selector, node health summary, confidence filter, and draggable splitter for custom layout.

### Authentication & Security
- **OIDC via Keycloak:** Web console protected by OpenID Connect authentication (Authorization Code + PKCE flow).
- **HTTPS:** Self-signed certificate for dev/test; production supports environment-variable-injected keystore password for CA-signed certificates.
- **Embedded MQTT CA:** Local certificate authority provisioned during first startup for mTLS node commissioning without external PKI infrastructure.

### Automated Test Coverage
- **Embedded Simulator:** Python/Java threat event generator producing realistic MQTT payloads (node heartbeats, threat classifications, azimuth/confidence vectors, speed).
- **Integration Tests:** Testcontainers-based test suite with PostgreSQL, Mosquitto, and WebSocket clients; validates end-to-end flows (ingestion → persistence → broadcast → frontend).
- **Unit Tests:** Core business logic coverage (threat detection, node lifecycle, CoT serialization).

### Node Lifecycle Management
- **Automated GC:** Scheduled job (30-second interval) detects stale nodes (last-seen > 3 minutes) and transitions them to OFFLINE state.
- **Event Auditing:** All state transitions logged to database for incident reconstruction.

## Next Steps (Immediate Roadmap)

### 1. Dynamic Configuration Service (Stage 13)
- Replace hardcoded parameters (multicast group/port, GC intervals, node thresholds) with a runtime configuration API.
- Expose configuration via dedicated REST endpoint and optional UI control panel.
- Enable operators to tune mesh parameters without redeployment.

### 2. Cloud Infrastructure Staging (Stage 14)
- Deploy gateway and dependencies (PostgreSQL, Keycloak, MQTT broker) to AWS or Azure cloud environment.
- Establish public staging URL for remote access and demo.
- Implement observability (structured logging, metrics via Micrometer, optional APM integration).

### 3. Live Demo Environment with Real Sensor Integration (Stage 15)
- Commission actual Onyx Grid edge-deployed sensor nodes in a controlled geographic region.
- Transition simulator to shadow mode (compare synthetic vs. real payloads for validation).
- Establish persistent live demo environment for stakeholder evaluation and operational training.

## Status

**Development Stage:** Stage 12 complete (UI enrichment, HTTPS, all core features operational). System ready for configuration service abstraction and cloud staging.

**Limitations (MVP Context):**
- All sensor telemetry currently synthetic (simulator-generated).
- Self-signed HTTPS certificate requires manual trust override in development.
- Keycloak provisioning requires manual initial setup.
- No native mobile client (web-only, works offline on cached state).

**Production Readiness:** With cloud deployment, real sensor integration, and dynamic configuration, system is feasible for small-scale tactical mesh networks (50–200 edge nodes per gateway).
